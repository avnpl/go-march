## Issue #1: Context Pointer Anti-Pattern

### Location
`repos/product_repo.go:61`

### The Problem
```go
// BAD: Passing context as a pointer
func (r pgProductRepo) UpdateByID(ctx *context.Context, p *models.UpdateProductReq) (models.Product, error)
```

### Why It's Wrong
In Go, `context.Context` is designed to be passed **by value**, never by pointer. The Go team designed it this way because:
- Context is meant to be immutable
- Passing pointers adds complexity and potential for bugs
- It's simply not the Go idiom

### How to Fix It
```go
// GOOD: Pass context by value
func (r pgProductRepo) UpdateByID(ctx context.Context, p *models.UpdateProductReq) (models.Product, error)
```

And update the call site in `services/product_service.go:64`:
```go
// Before (current):
res, err := s.repo.UpdateByID(&ctx, req)

// After:
res, err := s.repo.UpdateByID(ctx, req)
```

## Issue #2: Silent JSON Decode Failure

### Location
`main.go:81`

### The Problem
```go
// BAD: No error check - silently fails on malformed JSON
json.NewDecoder(r.Body).Decode(&params)
```

### Why It's Wrong
If a client sends invalid JSON to `/graphql`, `params.Query` becomes an empty string. The GraphQL handler then runs with no query, returning a confusing error. You'll spend hours debugging "why is my query not working" when the real issue is the JSON was malformed.

### How to Fix It
```go
// GOOD: Always check decode errors
var params struct {
    Query string `json:"query"`
}
if err := json.NewDecoder(r.Body).Decode(&params); err != nil {
    logger.Error("failed to decode GraphQL request", zap.Error(err))
    utils.SendJSONError(w, http.StatusBadRequest, "Invalid JSON")
    return
}

if params.Query == "" {
    utils.SendJSONError(w, http.StatusBadRequest, "Missing query")
    return
}
```

## Issue #3: Unchecked Error — ParseInt Result Overwritten

### Location
`api/rest/product_handler.go:75-76` (FetchProduct) and `api/rest/product_handler.go:150-151` (DeleteProduct)

### The Problem
```go
// BAD: ParseInt error is never checked — immediately overwritten by the next call
id, err := strconv.ParseInt(idStr, 10, 64)
prod, err := h.svc.GetProductByID(r.Context(), id)  // ← err from ParseInt is silently lost
```

The same pattern exists in `DeleteProduct`:
```go
id, err := strconv.ParseInt(idStr, 10, 64)
prod, err := h.svc.DeleteProduct(r.Context(), id)
```

### Why It's Wrong
Since `prod` is a new variable, Go allows the `:=` on the second line — but `err` is **overwritten** (not checked) from the first call. If `strconv.ParseInt` fails (e.g., user sends "abc" as ID), the code continues with a zero-value `id` (0). The service call then hits the database with ID 0, which either returns a confusing "not found" error or (worse) matches the wrong row.

**Note:** This is technically an overwritten error, not a shadowed variable (shadowing requires a nested scope). The practical impact is the same — the parse error is silently lost.

### How to Fix It
```go
// GOOD: Handle each error explicitly
id, err := strconv.ParseInt(idStr, 10, 64)
if err != nil {
    h.log.Error("invalid product ID", zap.String("id", idStr), zap.Error(err))
    utilErrs.SendJSONError(w, http.StatusBadRequest, "Invalid product ID format")
    return
}

prod, err := h.svc.GetProductByID(r.Context(), id)
if err != nil {
    // ... handle service error
}
```

Apply this fix in **both** `FetchProduct` and `DeleteProduct`.

## Issue #4: Wrong HTTP Status Codes

### Location
`api/rest/product_handler.go:87, 136, 162`

Three handlers use `http.StatusCreated` (201) when they should not:
- **Line 87** — `FetchProduct` (GET) returns 201 instead of 200
- **Line 136** — `UpdateProduct` (PATCH) returns 201 instead of 200
- **Line 162** — `DeleteProduct` (DELETE) returns 201 instead of 204

`FetchAllProducts` (line 99) correctly uses `http.StatusOK` — no change needed there.

### The Problem
```go
// BAD: Using StatusCreated (201) for GET, PATCH, and DELETE requests
w.WriteHeader(http.StatusCreated)  // This appears in FetchProduct, UpdateProduct, DeleteProduct
```

### Why It's Wrong
HTTP status codes have specific meanings:
- `200 OK` — Request succeeded, usually for GET/UPDATE
- `201 Created` — New resource was created (POST)
- `204 No Content` — Successful deletion (DELETE)
- `400 Bad Request` — Client sent bad data
- `404 Not Found` — Resource doesn't exist

Using wrong codes confuses API consumers and violates HTTP semantics.

### How to Fix It
```go
// GOOD: Use appropriate status codes
func (h ProductHandler) FetchProduct(w http.ResponseWriter, r *http.Request) {
    // ... fetch product ...
    w.WriteHeader(http.StatusOK)  // GET succeeded
    json.NewEncoder(w).Encode(prod)
}

func (h ProductHandler) UpdateProduct(w http.ResponseWriter, r *http.Request) {
    // ... update product ...
    w.WriteHeader(http.StatusOK)  // UPDATE succeeded
    json.NewEncoder(w).Encode(prod)
}

func (h ProductHandler) DeleteProduct(w http.ResponseWriter, r *http.Request) {
    // ... delete product ...
    w.WriteHeader(http.StatusNoContent)  // DELETE succeeded, no body to return
    // Don't encode JSON after 204 - there's no body!
}
```

## Issue #5: Incorrect Error Logging

### Location
`api/rest/product_handler.go:32, 46, 68, 94, 106, 120`

### The Problem
```go
// BAD: Wrapping error then losing the stack trace
h.log.Error(fmt.Errorf("error reading the request body : %w", err).Error(), zap.Error(err))
```

This does TWO wrong things:
1. `fmt.Errorf(...).Error()` converts the error to a string, losing the error chain
2. You're logging the error message as the log message AND the error field—redundant

This pattern appears in **six places**:
- Line 32 (`CreateProduct` — read body error)
- Line 46 (`CreateProduct` — decode error)
- Line 68 (`FetchProduct` — no ID)
- Line 94 (`FetchAllProducts` — extra bad: `fmt.Errorf("FetchAllProducts failed")` doesn't include `err` at all, so the original error is **completely lost**)
- Line 106 (`UpdateProduct` — read body error)
- Line 120 (`UpdateProduct` — decode error)

### How to Fix It
```go
// GOOD: Log the message and the error properly
h.log.Error("failed to read request body", zap.Error(err))

// If you need additional context:
h.log.Error("failed to parse request body",
    zap.String("content_type", r.Header.Get("Content-Type")),
    zap.Error(err),
)
```

## Issue #6: Dead Validation Tags

### Location
`models/models.go:21-24`

### The Problem
```go
// The tags say "required" but nothing validates them!
type CreateProductReq struct {
    Name  string  `json:"name" validate:"required"`
    Price float64 `json:"price" validate:"required"`
    Stock int     `json:"stock" validate:"required"`
}
```

The `validate:"required"` tags are useless—there's no validation library reading them.

### How to Fix It

**Option A: Manual validation (simple, no extra dependencies)**
```go
func ValidateCreateProductReq(req *CreateProductReq) error {
    if req.Name == "" {
        return errors.New("name is required")
    }
    if req.Price <= 0 {
        return errors.New("price must be greater than 0")
    }
    if req.Stock < 0 {
        return errors.New("stock cannot be negative")
    }
    return nil
}
```

**Option B: Use a validation library (if you want to learn)**
```go
import "github.com/go-playground/validator/v10"

// In handler:
var req models.CreateProductReq
if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
    // ...
}

validate := validator.New()
if err := validate.Struct(&req); err != nil {
    // Return validation errors to client
}
```

For this project, **Option A** is recommended—fewer dependencies, simpler code.

## Issue #7: String for Timestamps

### Location
`models/models.go:8-9, 17`

### The Problem
```go
// BAD: Using string for timestamps
type Product struct {
    CreatedAt string  `db:"created_at" json:"created_at"`
    UpdatedAt string  `db:"updated_at" json:"updated_at"`
}

type Orders struct {
    CreatedAt string `db:"order_time"`
}
```

Using `string` for timestamps causes:
- No type safety (can't do time operations)
- Format inconsistencies across the app
- Harder to serialize/deserialize correctly

### How to Fix It
```go
// GOOD: Use time.Time with proper JSON serialization
import "time"

type Product struct {
    ProductID  string    `db:"prod_id" json:"prod_id"`
    Name       string    `db:"prod_name" json:"prod_name"`
    Price      float64   `db:"price" json:"price"`
    Stock      int       `db:"stock" json:"stock"`
    CreatedAt  time.Time `db:"created_at" json:"created_at"`
    UpdatedAt  time.Time `db:"updated_at" json:"updated_at"`
}

type Orders struct {
    OrderID    string    `db:"order_id"`
    Quantity   int       `db:"quantity"`
    ProductID  string    `db:"product_id"`
    Amount     string    `db:"total_price"`
    CreatedAt  time.Time `db:"order_time"`
}
```

Go's `json.Marshal` automatically converts `time.Time` to RFC3339 format (`"2006-01-02T15:04:05Z07:00"`). If you need a different format, use a custom marshaler.

## Issue #8: .env Loaded on Every Call

### Location
`utils/utils.go:16`

### The Problem
```go
// BAD: Loads .env file on every single call!
func getEnvVar(key string) string {
    err := godotenv.Load(".env")  // File I/O every time
    if err != nil {
        log.Fatalln("Error loading .env file...")
    }
    return os.Getenv(key)
}
```

If you call `getEnvVar` 10 times, you load the .env file 10 times. Wasteful and slow.

### How to Fix It
```go
// GOOD: Load once at startup
package utils

var envOnce struct {
    loaded bool
}

func init() {
    // init() runs once when the package is first imported
    if err := godotenv.Load(".env"); err != nil {
        log.Println("Warning: .env file not loaded (using system env vars)")
    }
}

func getEnvVar(key string) string {
    return os.Getenv(key)
}
```

Or even better—in `main.go`, load it once:
```go
func main() {
    // Load .env at the very start
    if err := godotenv.Load(".env"); err != nil {
        log.Println("No .env file found, using system environment variables")
    }
    
    logger := utils.BuildLogger()
    // ...
}
```

## Issue #9: No Database Connection Pooling Config

### Location
`utils/utils.go:54-64`

### The Problem
```go
// BAD: No connection limits configured
func GetDBPoolObject(logger *zap.Logger) *sqlx.DB {
    dsn := getEnvVar("DB_URL")
    if dsn == "" {
        logger.Fatal("DB_URL not set")
    }
    db, err := sqlx.Connect("pgx", dsn)
    if err != nil {
        logger.Fatal("db connect failed", zap.Error(err))
    }
    return db  // Returns with default pool settings!
}
```

Without configuration, defaults may not suit production:
- Too many connections → database overload
- Too few connections → performance bottleneck
- Connections never expire → resource leaks

### How to Fix It
```go
// GOOD: Configure the connection pool
func GetDBPoolObject(logger *zap.Logger) *sqlx.DB {
    dsn := getEnvVar("DB_URL")
    db, err := sqlx.Connect("pgx", dsn)
    if err != nil {
        logger.Fatal("db connect failed", zap.Error(err))
    }

    // Configure connection pool
    db.SetMaxOpenConns(25)              // Max concurrent connections
    db.SetMaxIdleConns(10)              // Keep this many idle
    db.SetConnMaxLifetime(5 * time.Minute)  // Recycle connections
    
    // Verify connection works
    if err := db.Ping(); err != nil {
        logger.Fatal("db ping failed", zap.Error(err))
    }

    logger.Info("database connected", zap.Int("max_open", 25))
    return db
}
```

## Issue #10: Logging Request Bodies (Security)

### Location
`api/rest/product_handler.go:38-41` (CreateProduct) and `api/rest/product_handler.go:112-115` (UpdateProduct)

### The Problem
```go
// CAUTION: Can leak sensitive data!
h.log.Debug("raw body",
    zap.Int("length", len(bodyBytes)),
    zap.String("body", strings.ReplaceAll(reqString, " ", "")),
)
```

This appears in **two** handlers — `CreateProduct` and `UpdateProduct` — both logging the full request body.

Even removing spaces, request bodies may contain:
- Passwords and tokens
- PII (names, emails, addresses)
- Credit card numbers
- API keys

**Rule:** Never log request bodies in production unless explicitly required and audited.

### How to Fix It
```go
// Option A: Don't log the body at all (recommended)
h.log.Debug("received request to create product",
    zap.String("content_type", r.Header.Get("Content-Type")),
)

// Option B: Log only in development, never in production
if os.Getenv("LOG_LEVEL") == "debug" {
    h.log.Debug("raw body", zap.ByteString("body", bodyBytes))
}
```

### Location

## Issue #13: `errors.As` Used Where `errors.Is` Is Needed

### Location
`api/rest/product_handler.go:53, 79, 153`

### The Problem
```go
// BAD: errors.As is for type-asserting error types, not comparing sentinel values
if errors.As(err, &utilErrs.ErrConflict) {   // line 53
if errors.As(err, &sql.ErrNoRows) {          // line 79
if errors.As(err, &sql.ErrNoRows) {          // line 153
```

### Why It's Wrong
`errors.As` is designed to unwrap an error chain and extract a **target type** (e.g., `*MyCustomError`). It expects its second argument to be a pointer to the target type.

`ErrConflict` and `sql.ErrNoRows` are **sentinel values** (specific error instances), not types. The correct function for matching sentinel values is `errors.Is`.

Passing `&utilErrs.ErrConflict` (a `*error`) to `errors.As` may silently fail to match even when the error IS in the chain, because `errors.As` is doing a type assertion, not a value comparison.

### How to Fix It
```go
// GOOD: Use errors.Is for sentinel error values
if errors.Is(err, utilErrs.ErrConflict) {
    utilErrs.SendJSONError(w, http.StatusConflict, "")
    return
}

if errors.Is(err, sql.ErrNoRows) {
    utilErrs.SendJSONError(w, http.StatusNotFound, "Record with given ID not found")
    return
}
```

## Issue #14: Unclosed `*sqlx.Rows` — Connection Leak

### Location
`repos/product_repo.go:87-101`

### The Problem
```go
func (r pgProductRepo) UpdateByID(ctx *context.Context, p *models.UpdateProductReq) (models.Product, error) {
    // ...
    result, err := r.db.NamedQueryContext(*ctx, query, args)
    if err != nil {
        return models.Product{}, fmt.Errorf("product_repo.Update: %w", err)
    }

    if result.Next() {
        err := result.StructScan(&res)
        // ...
    }
    // ← result is NEVER closed!
    return res, nil
}
```

### Why It's Wrong
`NamedQueryContext` returns `*sqlx.Rows`, which holds an open database connection from the pool. If you don't call `result.Close()`, that connection is never returned to the pool. Under load, the pool runs out of connections and the application hangs or errors out.

### How to Fix It
```go
result, err := r.db.NamedQueryContext(*ctx, query, args)
if err != nil {
    return models.Product{}, fmt.Errorf("product_repo.Update: %w", err)
}
defer result.Close()  // Always close Rows!

if result.Next() {
    // ...
}
```

`defer result.Close()` must go immediately after the `nil` error check. This guarantees cleanup regardless of how the function returns.

## Issue #15: `pkg/errors` vs Standard Library `errors` Mismatch

### Location
`utils/errors.go:4` (definition) vs `api/rest/product_handler.go:7` (usage)

### The Problem
```go
// utils/errors.go — sentinel errors created with pkg/errors
import "github.com/pkg/errors"

var ErrConflict = errors.New("ErrConflict")
```

```go
// api/rest/product_handler.go — compared with stdlib errors
import "errors"

errors.As(err, &utilErrs.ErrConflict)
```

### Why It's Wrong
The sentinel errors are created with `github.com/pkg/errors.New()`, but the handler compares them using `errors.Is`/`errors.As` from the standard library `errors` package. While `pkg/errors` does implement the `Unwrap` interface for compatibility, mixing two error packages creates unnecessary confusion and a fragile dependency chain.

Since Go 1.13, the standard library `errors` package supports `Is`, `As`, `Unwrap`, and `fmt.Errorf` with `%w` — everything `pkg/errors` provides. There is no longer a reason to use `pkg/errors` in new Go code.

### How to Fix It
```go
// utils/errors.go — use standard library only
import "errors"

var (
    ErrConflict       = errors.New("ErrConflict")
    ErrInternal       = errors.New("Internal Error")
    ErrInvalidRequest = errors.New("Invalid Request")
    ErrRecordNotFound = errors.New("Record Not Found")
)
```

Then remove `github.com/pkg/errors` from `go.mod`.
